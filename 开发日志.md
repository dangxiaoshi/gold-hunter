## 2026-04-06

### 话术提示词排查结论

- 本轮只做技术排查和方案整理，不改业务代码。
- 当前聊天页实际走的是 `public/pages/chat.html` 的 `runDirectTactics()`，调用后端接口 `/api/ai/tactics-v1`，不是另一套较早的 `/api/ai/tactics` 简版逻辑。
- 中间栏的产品和阶段并不是没传过去，前端会把 `product` 和 `progressStage` 一起发给后端。
- 真正的问题在于：后端 prompt 没有把“人工选择的阶段”设成硬约束，而是把它和“模型自己判断客户真正想要什么”同时写进提示词，导致模型可以重新改判阶段。
- 这会出现一种典型现象：界面已手动选中“解决顾虑”，但模型仍可能回到“挖掘需求”的说法和打法。

### 已确认的关键错位

- 阶段控制权不清：
  当前 `tactics-v1` prompt 中虽然写了“客户处于 ${curr} 阶段”，但同时要求“先判断客户现在真正想要什么，再给话术”，等于把阶段决定权又交回给模型。
- 对话读取策略不统一：
  `分析` 接口只看最近 `30` 条，右侧自由问答只看最近 `20` 条聊天和最近 `20` 条 AI 历史，而“直接出击”会把当前客户的 `messages` 全量发给后端。
- 话术重复没有硬性约束：
  当前 prompt 没有明确要求“不要复用已经发过的话”，也没有程序级的相似度去重，所以模型很容易给出已经发过或高度相似的话术。
- 产品库和话术库来源不一致：
  产品库来自服务端 `data/products.json`，话术库却来自浏览器本地 `localStorage['gh_scripts_v5']`。这意味着不同浏览器、不同本地缓存状态下，模型拿到的话术样本可能不一致。
- 产品名存在别名错位风险：
  实际产品库里主打产品名是“播客变现私教”，但使用者日常口径常说“播客私教”。前端当前产品卡匹配逻辑只做 `===` 或 `includes`，可能出现产品已选中但详细产品卡没真正注入 prompt 的情况。

### 这次问题的根因判断

- 根因一不是“模型太笨”，而是“系统把该由程序控制的状态，又放回给模型自由判断”。
- 根因二不是“资料不够多”，而是“资料来源不统一、上下文边界不清、阶段优先级不够硬”。
- 根因三不是“话术库太少”，而是“当前的大 prompt 同时承担了状态识别、资料拼接、风格参考、生成下一句等多个职责，职责过载，容易漂移”。

### 参考成熟 AI 产品的做法

- 一般不会把所有能力都堆进一个大提示词里。
- 常见做法是分层：
  `状态层` 负责保存产品、阶段、顾虑、是否报过价、目标动作。
  `上下文层` 负责提供最近相关聊天，而不是永远喂全量。
  `检索层` 负责按阶段和问题类型召回产品卡、话术样本、知识片段。
  `约束层` 负责写死硬规则，例如“人工阶段优先于模型自判”“禁止复用已发原句”。
  `生成层` 只负责在这些约束下产出下一句最该发的话。
- 对销售场景来说，真正稳定的不是“万能聊天型 prompt”，而是“结构化状态 + 检索 + 受控生成”。

### 推荐路线总览

- 不建议一上来做重型多 Agent。
- 最推荐的顺序是：
  1. 先做非 Agent 的最小重构版
  2. 稳定后再升级为轻 Agent 两阶段
  3. 暂不考虑多 Agent 协作式架构

### 方案 A：非 Agent 最小重构

- 目标：
  先解决三个最核心的问题：阶段对齐、减少重复、减少跑偏。
- 核心思路：
  把“状态判断”和“话术生成”拆开，但仍然保持成一个轻量工作流，不引入 Agent 框架。

#### A1. 输入结构化

- 前端或后端在进入生成前，先整理出结构化输入：
  `product`
  `stage`
  `objection_type`
  `customer_status`
  `whether_priced`
  `last_sent_messages`
  `goal`

#### A2. 阶段改成硬约束

- 如果中间栏手动选择了阶段，模型必须严格服从。
- prompt 中应明确写成：
  “当前阶段为人工确认结果，不允许改判为其他阶段；如判断有偏差，只能在分析区提示，但生成的话术仍必须严格按当前阶段执行。”

#### A3. 统一上下文读取策略

- `分析`：最近 `20-30` 条。
- `出击`：最近 `12-20` 条相关聊天 + 最近 `3-5` 条我方已发消息。
- `自由问答`：最近 `10-20` 条聊天 + 最近 `10-20` 条 AI 历史。
- 核心原则：
  优先“最近且相关”，不要长期全量灌入。

#### A4. 拆出顾虑类型

- 建议至少拆成：
  `价格顾虑`
  `时间顾虑`
  `效果顾虑`
  `执行力顾虑`
  `信任顾虑`
  `时机顾虑`
- “解决顾虑”本身过大，不拆类型的话，模型容易回到泛泛而谈。

#### A5. 检索改为按条件召回

- 话术库和知识库不要整包塞给模型。
- 建议按：
  `产品 + 阶段 + 顾虑类型`
  召回 `3-5` 条最相关话术和少量产品卡/知识片段。
- 产品库决定“卖什么”，话术库决定“怎么说”，知识库决定“补什么事实和案例”。

#### A6. 做重复约束

- prompt 层要求：
  禁止直接复用最近已发送消息中的原句。
- 程序层建议增加一层相似度比对：
  如果候选回复和最近 `5-10` 条我方消息高度相似，则自动要求模型重写一次。

#### A7. 收紧输出形态

- 默认只输出：
  `判断`
  `推荐发送`
  `备选1`
  `备选2`
- 不再默认输出大段策略说明和培训式分析，重点是“能直接发”。

### 方案 B：轻 Agent 两阶段

- 适合在方案 A 稳定后升级。
- 不建议直接跳到多 Agent，自由协作太重，调试和成本都不划算。

#### B1. 诊断 Agent

- 只负责判断，不负责写话术。
- 输入：
  最近聊天、人工选择的产品和阶段、最近已发消息。
- 输出结构化 JSON，例如：
  `follow_manual_stage`
  `stage`
  `objection_type`
  `customer_intent`
  `next_goal`
  `must_avoid`

#### B2. 生成 Agent

- 只负责根据诊断结果生成 `1-3` 条候选话术。
- 不允许它再次改判阶段。

#### B3. 适用价值

- 好处：
  可解释、便于调试、便于后续扩展复盘与评分。
- 代价：
  比非 Agent 版本更复杂、更慢、成本更高。

### 为什么现在不建议多 Agent

- 当前问题的核心不是推理深度不够，而是工作流边界不清。
- 上多 Agent 容易带来：
  成本高、延迟高、输出不稳、调试困难、责任边界混乱。
- 对金币猎人二当前阶段来说，先把“状态控制权”收回来，比引入复杂 Agent 架构更重要。

### 推荐执行顺序

#### 第一阶段

- 把人工阶段改成硬约束。
- 统一各入口上下文读取条数和策略。
- 补“禁止复用已发原句”。
- 统一产品库、话术库、知识库的职责边界。

#### 第二阶段

- 增加顾虑类型识别。
- 检索逻辑改成 `产品 + 阶段 + 顾虑类型`。
- 输出缩短，优先生成“下一句最该发的话”。

#### 第三阶段

- 再考虑诊断 Agent。
- 再考虑候选回复评分、排序、相似成交案例召回等增强能力。

### 当前最值得优先验证的假设

- 先只做下面三件事，预计就能明显改善：
  1. 人工阶段强制优先
  2. 禁止复用已发原句
  3. 统一上下文读取边界
- 如果这三条落地后效果明显提升，说明现阶段的主要矛盾确实不在模型能力，而在工作流和状态控制。

### 最终建议

- 现在最合适的是：
  先上方案 A，预留升级到方案 B 的接口。
- 判断标准不是“是不是更聪明”，而是：
  能不能更稳、更可控、更容易调试、更适合销售场景。
- 对金币猎人二来说，短期内最重要的是把它从“一个大 prompt 硬扛全部逻辑”，收敛到“结构化状态 + 小范围检索 + 受控生成”。
